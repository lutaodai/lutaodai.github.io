Bypass OOM Error of Integrated Gradients

Integrated Gradients method [1] requires sampling points between the input and a baseline . As a result, the input to the module is expanded times, and so is the backward gradient calculation, which often leads to a Out-Of-Memory (OOM) errors. This post modifies the Captum's implementation on Integrated Gradients to address this issue. The main idea is to divide the -step approximation into mini-batches and evaluate them sequentially.

The modified function can be naturally extended to other gradient-based attribution methods (those with an n_steps argument).

Codes can be found on Github.


Integrated Gradients Approximation

Recall that the integral in Integrated Gradients is approximated by setting sampling points between the input data and the baseline :

This linearly scales up the memory usage in a single forward and a backward pass. For models associated with high feature space dimensions, such as 3D medical images, and/or for sequential models dealing with very long dependencies, OOM may occur even when the batch size is 1. The official Captum documentation proposes decreasing the value of , but it also acknowledges that it may lead to lower-quality approximations. (it also proposes to use the internal_batch_size argument, but that just further divides the current batch dimension into smaller chunks which are subsequently processed sequentially, therefore did not address the OOM problem when batch size is 1.)

The solution is to divide the -step approximation into manageable chunks and process these chunks sequentially. The final approximation will be aggregated on CPU.
